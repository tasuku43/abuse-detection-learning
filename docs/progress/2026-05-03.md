## 学んだこと

* notebook は scoring logic の置き場ではなく、評価結果を手で観察する作業台として使う
* `src/` の関数を notebook から呼ぶと、実装本体と観察作業を分けられる
* threshold sweep を表で見ると、precision と recall のトレードオフを追いやすい
* false positive / false negative を行として見ると、score の数字だけでは分からない改善候補を考えやすい
* 複数の scoring_fn を同じ fixture と同じ threshold で比較すると、評価 harness が関数差し替えの器になっていることが分かりやすい
* conservative な scoring は false positive を減らしやすいが、false negative が増えやすい
* recall 重視の scoring は false negative を減らしやすいが、false positive が残りやすい
* 特徴量を増やすと、同じ threshold でも scoring_fn ごとの差が見えやすくなる
* ML に進んでも、良い特徴量を定義する作業は残る
* 有効な特徴量は、仮説、探索的分析、モデルによる feature importance、error analysis を行き来しながら探す
* Phase 3 では、評価済みの feature row を前提にするのではなく、その feature row をどう作るかを考える
* ML baseline に進む前に、point-in-time feature row と teacher label の作り方を整理する必要がある
* 最初の ML は logistic regression に絞ると、rule-based scoring_fn との差分を観察しやすい
