## 学んだこと

* notebook は scoring logic の置き場ではなく、評価結果を手で観察する作業台として使う
* `src/` の関数を notebook から呼ぶと、実装本体と観察作業を分けられる
* threshold sweep を表で見ると、precision と recall のトレードオフを追いやすい
* false positive / false negative を行として見ると、score の数字だけでは分からない改善候補を考えやすい
* 複数の scoring_fn を同じ fixture と同じ threshold で比較すると、評価 harness が関数差し替えの器になっていることが分かりやすい
* conservative な scoring は false positive を減らしやすいが、false negative が増えやすい
* recall 重視の scoring は false negative を減らしやすいが、false positive が残りやすい
* 特徴量を増やすと、同じ threshold でも scoring_fn ごとの差が見えやすくなる
* ML に進んでも、良い特徴量を定義する作業は残る
* 有効な特徴量は、仮説、探索的分析、モデルによる feature importance、error analysis を行き来しながら探す
* Phase 3 では、評価済みの feature row を前提にするのではなく、その feature row をどう作るかを考える
* ML baseline に進む前に、point-in-time feature row と teacher label の作り方を整理する必要がある
* 最初の ML は logistic regression に絞ると、rule-based scoring_fn との差分を観察しやすい
* teacher label に自動検知システムの停止結果を混ぜると、既存 detector の判断を正解として学習してしまうリスクがある
* offline evaluation の feature row は `user_id + as_of_time` を先に決め、その時点より前の event だけで特徴量を作る
* dbt skeleton は実DB接続より先に、label source、evaluation target、point-in-time feature の責務を分けて読むために役立つ
* SQLite を小さな local warehouse として使うと、Snowflake / TD に接続せずに raw data から feature row を作る流れを再現できる
* `business_events` や `td_events` のような具体名より、まずは「ユーザー行動ログ」「アカウント属性」「オペレーター操作ログ」という抽象概念で理解した方が転用しやすい
* feature row は raw table から直接読む行ではなく、評価時点を決めて、その時点より前のeventだけを集計して作る
* SQLite の `datetime(as_of_time, '-24 hours')` と `event_time < as_of_time` を使うと、point-in-time feature の考え方を小さく確認できる
* human label source では `actor_type = 'human'` だけを使うことで、自動検知システムの判断を teacher label に混ぜない境界を表現できる
* 実運用ではユーザー属性やプロフィールは application DB、行動ログは warehouse に分かれやすい
* 学習用には単一SQLiteでもよいが、`app_*` と `warehouse_*` のように責務を分けると実践構成へ対応づけやすい
* status の現在値を使うと未来情報が混ざりやすいため、snapshot または history として point-in-time に読む必要がある
* 実務運用では feature rows と labels を分けて保存し、評価runのタイミングでjoinした方が、遅れて確定するhuman labelを扱いやすい
* precision / recall だけでは、どの行が誤っているかは分からない
* false positive / false negative を行として取り出すと、feature のどこを見直すべきか考えやすい
* score bucket を見ると、個別行を見る前に、誤りが高 score 側や低 score 側へ偏っているか確認できる
* error analysis は scoring logic ではなく、評価後の観察として分けると責務が整理しやすい
* error analysis helper は notebook で表として見て初めて学習効果が出る
