## 学んだこと

* 既存の評価基盤は、本番運用そのものではなく、scorer を育てるための土台である
* 本番スコアリング運用では、scorer は直接 action せず、ScoreResult を出すだけに保つ

* Mermaid では四角だけを並べるより、`[(...)]` をデータ、`{{...}}` を判断、`[/.../]` を観察、角丸を処理として使うと、図だけでも責務の違いを追いやすい

## Design Doc Refinement で学んだこと

* Scoring Interface は単なる Web API ではなく、スコア提供レイヤーの入口として捉えると責務が明確になる
* Candidate Generation Layer を分けると、Target Selector、Decision Policy、ActionCandidate が「候補を作る」責務としてまとまる
* Candidate Consumption Layer を分けると、Review Queue View と Action Worker がどちらも ActionCandidate を読む側だと理解しやすい
* Deployment Units やログ設計は完全に独立した章ではなく、各レイヤーに紐づく論点と、レイヤー横断の論点に分けると読みやすい
* ScoreResult はレスポンスオブジェクト、`score_results` は optional な scoring log と分けると、同期 API 連携と監査ログの両方を自然に表現できる
* 冒頭図では optional log まで描くと主張がぶれるため、ログは本文の実装詳細として説明する方がよい
* Candidate Consumption Layer の主入力は `score_results` ではなく `action_candidates` であるため、ログ設計でもその読み手を混同しない方がよい
* `action_candidates` は生成履歴であるだけでなく、後続システムを駆動する operational dataset として扱う方が責務が明確になる
* 評価基盤もレイヤー化すると、運用スコアリングとの共通部分が `entity_id + as_of_time -> Feature Builder -> feature row -> Scorer -> score` だと見えやすい
* 評価基盤は score の後に `score + label -> metrics / improvement` へ進み、運用スコアリングは `ScoreResult -> Decision Policy -> ActionCandidate` へ進む
* feature row をオンメモリの scorer 入力として捉えると、dbt / SQLite / fixture は本番Feature Storeではなく、Feature Builder のロジックや出力を観察する教材として説明できる
