## 学んだこと

* このリポジトリでは、feature generation、scoring function、evaluation harness を分けて考える
* 最初は実 DB や ML framework ではなく、合成 CSV とルールベース scoring_fn で評価の流れを理解する
* progress を日別ファイルにすると、「昨日やったこと」などをピンポイントに読みやすい
* ルート直下は入口のドキュメントに絞り、進捗管理系は `docs/` に寄せると見通しがよい
* 実装手順だけでなく、各ステップで得られる学びを説明することが重要
* 「続きから始めよう」で再開できるようにするには、git status、tasks、progress、未コミット差分を見る手順を明記しておくとよい
* `scoring_fn` を純粋関数にしておくと、DB やファイル入出力なしでテストしやすい
* threshold=80 の false positives / false negatives を fixture で意図的に作ると、error analysis の入口を理解しやすい
* metrics は score 計算と分けることで、scoring rule を変えても precision / recall 計算を再利用できる
* 「Phase N をやる」と言うだけでは、学習用プロジェクトで必要な設計意図の説明として不足する
* 実装前説明は個人の注意力に頼らず、チェックリストと workflow に明文化しておく必要がある
* Python では型ヒントとして `TypedDict` を使うと、辞書形式の feature row が期待するキーを表現できる
* 処理の流れを Mermaid 図にすると、feature row、scoring、metrics、error analysis の責務分離を追いやすい
